# Self-Hosting Overview

Self-hosting runs the same core Planisfy application code on infrastructure you operate. It is useful for development, demos, private deployments, and small production installs where you own the runtime services.

## What the stack runs

The Docker Compose stack includes:

- API, Console, Admin, Docs, and Marketing apps.
- `worker-geodata` for uploads, imports, tiling, and artifact processing.
- Postgres and Redis.
- Martin for tiles and glyphs.
- Valhalla for routing.
- Pelias API and Elasticsearch for geocoding.
- Local elevation service.
- Static renderer.
- MinIO when the `with-minio` profile is enabled.

Optional profiles add Traefik, tile-worker, Pelias fixture import jobs, and the self-host supervisor.

## What you operate

With `DEPLOYMENT_MODE=self_host`, you are responsible for:

- Database and Redis availability.
- Object storage and artifact retention.
- PMTiles, fonts, Valhalla graph data, Pelias data, DEM files, and static renderer readiness.
- Backups, restores, upgrades, monitoring, and log retention.
- External root agents if you run regional or planet-scale builds.

Hosted checkout, hosted billing portal actions, managed subscription feature gates, and hosted plan quotas are disabled. Usage screens still show operational activity.

## Recommended path

1. Start with [Docker Compose](/docs/self-hosting/docker).
2. Review [configuration](/docs/self-hosting/configuration), especially URLs, secrets, and storage.
3. Add required [data sources](/docs/self-hosting/data-sources).
4. Create the first Console account.
5. Run health, preflight, and smoke checks.
6. Add [external compute](/docs/self-hosting/external-compute) only after the base stack is healthy.

## Dataset expectations

Planisfy does not ship production map datasets. A clean stack can boot with degraded service checks until compatible files or external providers are configured. Missing data should not prevent Console/API startup, but it will affect maps, geocoding, routing, elevation, or static rendering.
