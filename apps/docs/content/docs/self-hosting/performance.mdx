## Caching Strategies

### 1. CDN Caching

Use Cloudflare or similar for edge caching:

```yaml
# Cloudflare configuration
cache_rules:
  - url: "*.tiles"
    cache_ttl: 604800  # 7 days

  - url: "*.fonts"
    cache_ttl: 31536000  # 1 year

  - url: "*.sprites"
    cache_ttl: 604800  # 7 days
```

**Cache rules:**
- Static assets: 1 year
- Tiles: 12 hours (browser) / 7 days (CDN)
- Geocoding: 1 hour
- Routing: 5 minutes

### 2. Redis Caching

Add Redis cache for frequently accessed data:

```yaml
services:
  redis:
    image: redis:alpine
    command: redis-server --maxmemory 2gb --maxmemory-policy allkeys-lru
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
```

**Martin with Redis:**

```yaml
# martin-config.yaml
cache:
  type: redis
  redis_url: redis://redis:6379
  ttl: 3600  # 1 hour
```

### 3. Application-Level Caching

```javascript
// Node.js example with Node-cache
const NodeCache = require('node-cache');
const styleCache = new NodeCache({ stdTTL: 3600 }); // 1 hour

async function getCachedStyle(styleId) {
  let style = styleCache.get(styleId);

  if (!style) {
    style = await fetchStyle(styleId);
    styleCache.set(styleId, style);
  }

  return style;
}
```

## Database Optimization

### PostgreSQL Tuning

#### `postgresql.conf`

```ini
# Connection settings
max_connections = 200
shared_buffers = 4GB
effective_cache_size = 12GB
work_mem = 64MB
maintenance_work_mem = 512MB

# WAL settings
wal_buffers = 64MB
checkpoint_completion_target = 0.9
max_wal_size = 4GB

# Query optimization
random_page_cost = 1.1
effective_io_concurrency = 200

# Logging
log_min_duration_statement = 1000  # Log slow queries
```

#### Connection Pooling

```yaml
# martin-config.yaml
sources:
  postgis:
    type: postgresql
    connection_string: postgresql://planisfy:pass@postgres:5432/tiles
    max_connections: 50
    pool_idle_timeout: 30
    pool_max_lifetime: 3600
```

### Indexing

Add indexes for common queries:

```sql
-- Index on tile geometry
CREATE INDEX idx_tiles_geometry ON tiles USING GIST(geometry);

-- Index on zoom level
CREATE INDEX idx_tiles_zoom ON tiles(zoom);

-- Composite index
CREATE INDEX idx_tiles_zoom_xy ON tiles(zoom, x, y);
```

## Tile Server Optimization

### Martin Performance

#### Worker Threads

```yaml
# martin-config.yaml
server:
  worker_threads: 8  # Match CPU cores
```

#### Tile Size Optimization

```yaml
# Generate tiles with appropriate size
# Smaller tiles = faster transfer but more requests
# Larger tiles = fewer requests but more data

# For high zoom (14+): Use 256x256
# For low zoom (0-10): Use 512x512
```

#### Compression

```yaml
compression: true
compression_level: 6  # Balance CPU vs compression
```

### Horizontal Scaling

```yaml
# docker-compose.yml
services:
  martin:
    image: maplibre/martin
    deploy:
      replicas: 3
    environment:
      - CACHE_TYPE=redis
      - REDIS_URL=redis://redis:6379
    depends_on:
      - redis
```

**Load balancer configuration:**

```nginx
upstream martin {
    least_conn;  # Route to least busy server

    server martin_1:3000 max_fails=3 fail_timeout=30s;
    server martin_2:3000 max_fails=3 fail_timeout=30s;
    server martin_3:3000 max_fails=3 fail_timeout=30s;
}
```

## Horizontal Scaling

### Kubernetes Deployment

```yaml
# martin-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: martin
spec:
  replicas: 3
  selector:
    matchLabels:
      app: martin
  template:
    metadata:
      labels:
        app: martin
    spec:
      containers:
      - name: martin
        image: maplibre/martin:latest
        ports:
        - containerPort: 3000
        resources:
          requests:
            cpu: 1000m
            memory: 2Gi
          limits:
            cpu: 2000m
            memory: 4Gi
---
apiVersion: v1
kind: Service
metadata:
  name: martin
spec:
  selector:
    matchLabels:
      app: martin
  ports:
  - port: 3000
  type: LoadBalancer
```

### Auto-Scaling

```yaml
# autoscaler.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: martin-autoscaler
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: martin
  minReplicas: 3
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```


## Troubleshooting Performance Issues

### Slow Tile Loading

**Diagnose:**

```bash
# Check Martin metrics
curl http://localhost:3005/metrics

# Check PostgreSQL connections
docker exec postgres psql -U planisfy -d tiles \
  -c "SELECT count(*) FROM pg_stat_activity;"
```

**Solutions:**
- Increase worker threads
- Add caching layer
- Scale horizontally
- Optimize database queries

### High Memory Usage

**Diagnose:**

```bash
# Check memory usage
docker stats

# Check Redis memory
docker exec redis redis-cli INFO memory
```

**Solutions:**
- Reduce tile cache size
- Limit connections
- Add swap space
- Scale vertically

### Slow Geocoding

**Diagnose:**

```bash
# Check Elasticsearch cluster health
curl http://localhost:9200/_cluster/health

# Check index stats
curl http://localhost:9200/pelias/_stats
```

**Solutions:**
- Increase Elasticsearch heap
- Optimize index settings
- Reduce search scope
- Use fewer data sources

### Routing Delays

**Diagnose:**

```bash
# Check Valhalla logs
docker compose logs valhalla --tail=100

# Test route calculation time
time curl "http://localhost:8002/route?locations=-122.42,37.78;-122.45,37.75"
```

**Solutions:**
- Increase Valhalla concurrency
- Pre-load tiles
- Reduce service limits
- Use faster costing model

## Best Practices

### 1. Profile Before Optimizing

```bash
# Profile Martin
go tool pprof -http=:3000 http://localhost:3000/debug/pprof/
```

### 2. Monitor All Components

```bash
# Collect metrics from all services
prometheus.yml:
  scrape_configs:
    - job_name: 'martin'
    - job_name: 'pelias'
    - job_name: 'valhalla'
    - job_name: 'postgres'
```

### 3. Use Appropriate Cache TTLs

```yaml
cache:
  tiles: 43200      # 12 hours
  styles: 3600      # 1 hour
  glyphs: 31536000  # 1 year
  sprites: 604800   # 7 days
```

### 4. Scale Horizontally First

Scale horizontally before vertically:

- Add more instances (cheaper)
- Use load balancer
- Only then scale vertically

### 5. Test at Scale

```bash
# Simulate production load
locust --users=1000 --spawn-rate=100
```

### 6. Set Up Alerts Early

Configure alerts before issues occur:

```yaml
alerts:
  - name: HighLatency
    condition: latency > 500ms
    action: Auto-scale cluster
```
