# Rate Limits & Quotas Guide

This guide explains Planisfy's rate limits, how to monitor your usage, and strategies for working within quotas effectively.

---

## Rate Limit Overview

Planisfy uses rate limits to ensure fair resource allocation and system stability. Each API plan has specific request quotas measured in **requests per minute (RPM)** and **requests per day (RPD)**.

When you exceed a rate limit, you'll receive an **HTTP 429 Too Many Requests** response until the limit resets.

<Callout type="info" title="Rate Limit Types">
  Planisfy enforces both <strong>rate limits</strong> (burst protection per
  minute) and <strong>quotas</strong> (daily usage limits).
</Callout>

## Rate Limit Headers

Every API response includes headers indicating your current rate limit status:

### Header Fields

| Header                  | Description                          | Example      |
| ----------------------- | ------------------------------------ | ------------ |
| `X-RateLimit-Limit`     | Requests allowed per window          | `1000`       |
| `X-RateLimit-Remaining` | Requests remaining in current window | `947`        |
| `X-RateLimit-Reset`     | Unix timestamp when limit resets     | `1709673600` |

### Reading Rate Limit Headers

```javascript
const response = await fetch(
  "https://api.planisfy.com/styles/v1/planisfy/basic",
);

console.log("Limit:", response.headers.get("X-RateLimit-Limit"));
console.log("Remaining:", response.headers.get("X-RateLimit-Remaining"));
console.log(
  "Reset:",
  new Date(response.headers.get("X-RateLimit-Reset") * 1000),
);
```

**Output:**

```
Limit: 1000
Remaining: 947
Reset: 2024-03-05T12:00:00.000Z
```

### Server-Side Logging

Log rate limit status to monitor usage:

```javascript
function logRateLimit(response) {
  const remaining = response.headers.get("X-RateLimit-Remaining");
  const limit = response.headers.get("X-RateLimit-Limit");
  const percentUsed = (((limit - remaining) / limit) * 100).toFixed(1);

  if (percentUsed > 80) {
    console.warn(`⚠️ Rate limit at ${percentUsed}% capacity`);
  }
}
```

## HTTP 429: Too Many Requests

When you exceed a rate limit, you'll receive a 429 response:

### Response Format

```json
{
  "message": "Rate limit exceeded",
  "code": "rate_limit_exceeded",
  "retry-after": 60,
  "request_id": "req_abc123xyz"
}
```

### Response Headers

| Header              | Description                      |
| ------------------- | -------------------------------- |
| `Retry-After`       | Seconds to wait before retrying  |
| `X-RateLimit-Reset` | Unix timestamp when limit resets |

### Handling 429 Responses

```javascript
async function fetchWithRetry(url, options = {}, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    const response = await fetch(url, options);

    if (response.status === 429) {
      const retryAfter = parseInt(response.headers.get("Retry-After")) || 5;
      console.log(`Rate limited. Waiting ${retryAfter} seconds...`);

      await new Promise((resolve) => setTimeout(resolve, retryAfter * 1000));
      continue;
    }

    return response;
  }

  throw new Error("Max retries exceeded");
}
```

## Monitoring Your Usage

### Account Dashboard

View your usage in the Planisfy dashboard:

- **Real-time requests** per minute
- **Daily usage** trends
- **Per-endpoint** breakdown
- **Top IP addresses** by request volume

### Programmatic Monitoring

Create a usage monitoring endpoint:

```javascript
// Get current usage statistics
async function getUsageStats() {
  const response = await fetch("https://api.planisfy.com/usage");
  const stats = await response.json();

  return {
    requestsToday: stats.requests.today,
    requestsRemaining: stats.quota.daily - stats.requests.today,
    percentUsed: ((stats.requests.today / stats.quota.daily) * 100).toFixed(1),
  };
}
```

### Set Up Alerts

Configure alerts to notify you when:

| Alert Type           | Trigger                     | Action             |
| -------------------- | --------------------------- | ------------------ |
| Daily quota warning  | 80% of daily quota used     | Email notification |
| Daily quota critical | 95% of daily quota used     | SMS + email        |
| Rate limit hit       | 3+ rate limits in 5 minutes | PagerDuty alert    |
| Unusual spike        | 2x normal traffic           | Security alert     |

```javascript
// Alerting example
async function checkUsageAndAlert(stats) {
  if (stats.percentUsed > 80) {
    await sendAlert({
      severity: "warning",
      message: `API usage at ${stats.percentUsed}%`,
      metrics: stats,
    });
  }
}
```

## Strategies for Handling Limits

### 1. Client-Side Caching

Reduce API calls by caching responses:

```javascript
const cache = new Map();

async function getCachedStyle(styleId) {
  if (cache.has(styleId)) {
    return cache.get(styleId);
  }

  const response = await fetch(
    `https://api.planisfy.com/styles/v1/planisfy/${styleId}`,
  );

  const style = await response.json();

  // Cache for 1 hour
  cache.set(styleId, style);
  setTimeout(() => cache.delete(styleId), 60 * 60 * 1000);

  return style;
}
```

### 2. Request Debouncing

Debounce rapid user actions:

```javascript
function debounce(func, wait) {
  let timeout;
  return function (...args) {
    clearTimeout(timeout);
    timeout = setTimeout(() => func.apply(this, args), wait);
  };
}

// Debounce geocoding calls
const debouncedGeocode = debounce(geocodeAddress, 500);

searchInput.addEventListener("input", (e) => {
  debouncedGeocode(e.target.value);
});
```

### 3. Batch Requests

Combine multiple requests when possible:

```javascript
// ❌ Multiple geocoding calls
for (const address of addresses) {
  await geocode(address);
}

// ✅ Batch geocoding
const results = await batchGeocode(addresses); // Up to 50 at once
```

### 4. Conditional Requests

Use ETags for conditional requests:

```javascript
let cachedETag = null;

async function getStyleWithETag(styleId) {
  const headers = {};
  if (cachedETag) {
    headers["If-None-Match"] = cachedETag;
  }

  const response = await fetch(url, { headers });

  cachedETag = response.headers.get("ETag");

  if (response.status === 304) {
    return cachedStyle; // Not modified
  }

  return await response.json();
}
```

### 5. Priority Queuing

Implement a queue for important requests:

```javascript
class RequestQueue {
  constructor(maxConcurrent = 5) {
    this.queue = [];
    this.active = 0;
    this.maxConcurrent = maxConcurrent;
  }

  async add(fn) {
    return new Promise((resolve, reject) => {
      this.queue.push({ fn, resolve, reject });
      this.process();
    });
  }

  async process() {
    if (this.active >= this.maxConcurrent || this.queue.length === 0) {
      return;
    }

    this.active++;
    const { fn, resolve, reject } = this.queue.shift();

    try {
      const result = await fn();
      resolve(result);
    } catch (error) {
      reject(error);
    } finally {
      this.active--;
      this.process();
    }
  }
}

// Usage
const queue = new RequestQueue(10); // Max 10 concurrent requests

for (const task of tasks) {
  await queue.add(() => fetch(task.url));
}
```

## Reducing Tile Requests

Tiles are typically the highest-volume requests. Optimize tile loading:

### 1. Use Appropriate Zoom Levels

```javascript
// Don't load unnecessary high-zoom tiles
map.on("zoomend", () => {
  const zoom = map.getZoom();

  // Clear layers beyond zoom 15
  if (zoom > 15) {
    console.warn("High zoom level - consider limiting detail");
  }
});
```

### 2. Limit Visible Area

```javascript
// Only load tiles for viewport
map.setMaxBounds([
  [-122.5, 37.7], // Southwest
  [-122.3, 37.8], // Northeast
]);
```

### 3. Enable Tile Caching

```javascript
const map = new maplibregl.Map({
  container: "map",
  style: "https://api.planisfy.com/styles/v1/planisfy/basic",
  // MapLibre automatically caches tiles
  crossOrigin: "anonymous",
});
```

### 4. Use Vector Tiles Instead of Raster

Vector tiles are more efficient:

- **Smaller file size** (typically 10-50KB vs 100KB+ for raster)
- **Client-side styling** (no multiple style requests)
- **Better caching** (one tile for all zoom levels)

## Requesting a Quota Increase

If you consistently hit rate limits, consider upgrading:

### When to Request an Increase

- **Consistently exceeding** daily quotas
- **Legitimate traffic growth** (not abuse/bugs)
- **Production application** with real users
- **Seasonal spikes** (planned in advance)

### How to Request

1. **Check your usage** in the dashboard
2. **Document your use case**:
   - Average daily requests
   - Peak traffic times
   - Growth projections
3. **Contact**: sales@planisfy.com
4. **Include**:
   - Account ID
   - Current plan
   - Desired limits
   - Business justification

await supportRequest({

## Rate Limiting Best Practices

### Do's ✅

- **Monitor usage** regularly via dashboard
- **Implement caching** at multiple levels
- **Use retries** with exponential backoff
- **Set up alerts** before hitting limits
- **Optimize tile loading** for maps
- **Batch requests** when possible

### Don'ts ❌

- **Don't poll** APIs continuously (use webhooks if available)
- **Don't ignore** 429 responses
- **Don't hardcode** API keys in frontend
- **Don't request** unnecessary data
- **Don't cache** beyond the recommended TTL
- **Don't abuse** free tier for production

## Troubleshooting

### Sudden Spike in Requests

**Symptoms:** Unexpected increase in usage

**Investigation:**

1. Check dashboard for top requesting IPs
2. Review application logs for loops/errors
3. Verify caching is working correctly
4. Check for unauthorized API key usage

**Solution:**

- Rotate compromised API keys
- Fix application bugs causing excessive requests
- Implement stricter rate limiting

### Consistently Hitting Limits

**Symptoms:** Regularly exceeding daily quota

**Solution:**

1. **Optimize** your application (caching, batching)
2. **Upgrade** to a higher plan
3. **Contact support** for custom limits

### Development vs Production Usage

**Problem:** Development and production share same quota

**Solution:**
Use separate API keys for different environments:

```javascript
// Development
const devKey = "pk_dev_environment_key";

// Staging
const stagingKey = "pk_staging_environment_key";

// Production
const prodKey = "pk_production_environment_key";
```

Each can have separate quotas and monitoring.
