# Authentication Guide

This guide covers how to securely authenticate with the Planisfy API using API keys, manage tokens across different environments, and implement security best practices.

---

## API Key Overview

Planisfy uses API keys to authenticate protected service APIs and associate
requests with your account. Published public map assets can load anonymously;
private resources and service endpoints require a valid API key.

### API Key Format

Planisfy API keys start with `pk_`:

```
pk_0123456789abcdef_0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef
```

<Callout type="info" title="Key Format">
  The Console shows the full key only once when it is created. Store it
  securely, then use it in the <code>X-API-Key</code> header for protected
  requests.
</Callout>

---

## Token Types

### Publishable API Keys

**Prefix:** `pk_`

**Purpose:** Client-side applications (web, mobile)

**Use cases:**

- Rendering maps in a browser
- Mobile app map display
- Public API requests

**Limitations:**

- Cannot access account management endpoints
- Should be used for read operations only
- Visible to users in the browser/app

**Example:**

```
pk_0123456789abcdef_0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef
```

<Callout type="warn" title="Security Warning">
  API keys are secrets. If a key is leaked, revoke it immediately in your
  account dashboard and create a replacement key.
</Callout>

## Using API Keys in Requests

### Public Map Asset URLs

Published public styles can be loaded directly by MapLibre:

```javascript
// MapLibre GL JS
const map = new maplibregl.Map({
  container: "map",
  style: `https://api.planisfy.com/styles/v1/planisfy/basic`,
  center: [-122.4194, 37.7749],
  zoom: 12,
});
```

```bash
# cURL
curl "https://api.planisfy.com/styles/v1/planisfy/basic"
```

### API Key Header

Use the `X-API-Key` header for protected Console and service APIs:

```javascript
// Fetch API
fetch("https://api.planisfy.com/geocoding/v1/search?text=Stuttgart", {
  headers: {
    "X-API-Key": "pk_your_key_here",
  },
});
```

```bash
# cURL
curl "https://api.planisfy.com/geocoding/v1/search?text=Stuttgart" \
  -H "X-API-Key: pk_your_key_here"
```

**Advantages:**

- Cleaner URLs (no long query strings)
- Standard HTTP authentication pattern
- Matches the Planisfy API key middleware

### Environment Variables

Store API keys in environment variables for server-side apps:

```bash
# .env file
PLANISFY_API_KEY=pk_your_key_here
```

```javascript
// Node.js
import "dotenv/config";

const apiKey = process.env.PLANISFY_API_KEY;

fetch("https://api.planisfy.com/geocoding/v1/search?text=Stuttgart", {
  headers: { "X-API-Key": apiKey },
});
```

```python
# Python
import os
from dotenv import load_dotenv

load_dotenv()

api_key = os.getenv('PLANISFY_API_KEY')
```

<Callout type="tip" title="Never Commit .env Files">
  Add <code>.env</code> to your <code>.gitignore</code> file to prevent
  accidentally committing secrets.
</Callout>

## Token Scopes

Scopes restrict what a token can do. Assign only the scopes needed for your application.

### Available Scopes

| Scope           | Description               | Use Case             |
| --------------- | ------------------------- | -------------------- |
| `tiles`         | Access vector tiles       | Map rendering        |
| `styles`        | Read style JSON           | Custom style loading |
| `fonts`         | Access font glyphs        | Text rendering       |
| `sprites`       | Access sprite images      | Icon rendering       |
| `geocoding`     | Forward/reverse geocoding | Address search       |
| `directions`    | Turn-by-turn routing      | Navigation           |
| `optimization`  | Route optimization        | Multi-stop routing   |
| `isochrone`     | Reachability analysis     | Service areas        |
| `dataset`       | Read datasets             | Custom data access   |
| `dataset:write` | Modify datasets           | Data management      |
| `styles:write`  | Create/update styles      | Style management     |

**Default scopes:** Public tokens include `tiles`, `styles`, `fonts`, `sprites`, `geocoding`, `directions`

### Scoped Token Example

Create a token for a geocoding-only application:

```bash
curl "https://api.planisfy.com/tokens/v2" \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{
    "scopes": ["geocoding"],
    "note": "Geocoding app - production"
  }'
```

## Securing API Keys in Frontend Applications

### 1. Keep Keys Scoped And Restricted

Frontend code is publicly visible. Use narrowly scoped `pk_` keys and domain
restrictions for browser applications:

```javascript
// ✅ Correct - Public token
const apiKey = "pk_your_key_here";

// ❌ Wrong - Never use secret tokens in frontend
const apiKey = "pk_unrestricted_production_key";
```

### 2. Domain Restrictions

Restrict API keys to specific domains in your account dashboard:

| Key             | Allowed Domains                                  |
| --------------- | ------------------------------------------------ |
| Production key  | `https://yourapp.com`, `https://www.yourapp.com` |
| Development key | `http://localhost:3000`, `http://127.0.0.1:8080` |

If a request comes from an unauthorized domain, it will be rejected.

### 3. Referer Limiting

When you can't use domain restrictions, use URL referer checking:

```javascript
// In your dashboard, set allowed referers:
// https://yourapp.com/*
```

<Callout type="warn" title="Referer Headers Can Be Spoofed">
  Referer restrictions are a first line of defense, not a complete security
  solution. Always combine with other measures.
</Callout>

### 4. Separate Keys Per Environment

Use different API keys for development, staging, and production:

| Environment | Key              | Scopes         | Domain Restrictions   |
| ----------- | ---------------- | -------------- | --------------------- |
| Development | `pk_dev_key`     | All scopes     | `localhost`           |
| Staging     | `pk_staging_key` | All scopes     | `staging.yourapp.com` |
| Production  | `pk_prod_key`    | Limited scopes | `yourapp.com`         |

This limits the impact if one key is compromised.

## Securing API Keys in Backend Applications

### 1. Environment Variables

Store secrets in environment variables, not in code:

```javascript
// ❌ Wrong - Hardcoded secret
const apiKey = "pk_your_key_here";

// ✅ Correct - Environment variable
const apiKey = process.env.PLANISFY_API_KEY;
```

### 2. Secret Management Services

For production, use a secret manager:

- **AWS:** Secrets Manager, Parameter Store
- **Google Cloud:** Secret Manager
- **Azure:** Key Vault
- **HashiCorp:** Vault

### 3. .gitignore Patterns

Ensure secrets are never committed:

```
# .gitignore
.env
.env.local
.env.production
config/secrets.json
**/secrets/**
```

### 4. Pre-Commit Hooks

Add a pre-commit hook to detect accidental commits:

```bash
# .git/hooks/pre-commit
if git diff --cached --name-only | grep "\.env"; then
  echo "Warning: .env file detected in commit!"
  echo "Remove .env files before committing."
  exit 1
fi
```

## API Key Rotation Best Practices

### Regular Rotation Schedule

Rotate API keys periodically to limit exposure:

| Key Type         | Rotation Frequency |
| ---------------- | ------------------ |
| Production keys  | Every 90 days      |
| Development keys | Every 30 days      |
| Compromised keys | Immediately        |

### Rotation Steps

1. **Create a new key** in your account dashboard
2. **Update your application** with the new key
3. **Deploy the update** to all environments
4. **Monitor for errors** for 24-48 hours
5. **Delete the old key** once everything is working

### Zero-Downtime Rotation

For production systems, use a rotation window:

```javascript
// Support multiple keys during rotation
const keys = {
  current: "pk_new_key_here",
  legacy: "pk_old_key_here", // Keep for fallback
};

async function makeRequest(url) {
  // Try current token first
  try {
    return await fetch(url, { headers: { "X-API-Key": keys.current } });
  } catch (error) {
    // Fallback to legacy key
    return await fetch(url, { headers: { "X-API-Key": keys.legacy } });
  }
}
```

### Automated Rotation Alerts

Set up monitoring to alert you when keys need rotation:

```javascript
// Check key age
const keyCreatedAt = new Date("2025-01-01");
const maxAge = 90 * 24 * 60 * 60 * 1000; // 90 days in ms

if (Date.now() - keyCreatedAt.getTime() > maxAge) {
  console.warn("API key needs rotation!");
  // Send alert to your monitoring system
}
```

## Monitoring API Key Usage

### View Usage Statistics

Check your account dashboard for:

- Requests per day
- Requests per minute
- Breakdown by API endpoint
- Top requesting IPs/domains

### Identify Abnormal Patterns

Look for:

- **Spikes in usage** → Possible token leak or abuse
- **Requests from unknown IPs** → Token may be compromised
- **Requests to unused endpoints** → Unauthorized access

### Set Up Alerts

Configure alerts for:

- Daily quota exceeded (80%, 90%, 100%)
- Rate limit hit (more than X times per hour)
- New requesting IP addresses
- Unusual endpoint access

## Self-Hosted Authentication

When self-hosting Planisfy, you have additional authentication options:

### 1. No Authentication (Internal)

For internal/private deployments:

```yaml
# martin-config.yaml
auth:
  enabled: false
```

### 2. API Key Middleware

Implement custom authentication:

```javascript
// Express.js middleware
const validKeys = new Set(["pk_key_1", "pk_key_2"]);

function validateApiKey(req, res, next) {
  const key = req.headers["x-api-key"];

  if (!validKeys.has(key)) {
    return res.status(401).json({ error: "Invalid API key" });
  }

  next();
}

app.use("/tiles", validateApiKey);
```

### 3. OAuth 2.0 / JWT

For enterprise deployments:

```javascript
// JWT validation
const jwt = require("jsonwebtoken");

function validateJwt(req, res, next) {
  const token = req.headers.authorization?.replace("Bearer ", "");

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded;
    next();
  } catch (error) {
    res.status(401).json({ error: "Invalid token" });
  }
}
```

### 4. Reverse Proxy Authentication

Use Nginx or Traefik for authentication:

```nginx
# nginx.conf
location /api/ {
  auth_request /auth;
  proxy_pass http://planisfy-backend;
}

location = /auth {
  internal;
  proxy_pass http://auth-service/validate;
}
```

## Troubleshooting

### "Invalid API Key" Error

**Problem:** API returns 401 Unauthorized

**Solutions:**

1. Verify the key starts with `pk_`
2. Check for typos in the key
3. Ensure the key hasn't been deleted
4. Verify domain restrictions (if configured)

### "Forbidden" Error

**Problem:** API returns 403 Forbidden

**Solutions:**

1. Check if the API key has required scopes
2. Verify rate limits haven't been exceeded
3. Ensure account is in good standing

### API Key Not Working in Production

**Problem:** API key works locally but fails in production

**Solutions:**

1. Check domain restrictions (add production domain)
2. Verify environment variables are set
3. Check for HTTPS requirements (some tokens require HTTPS)
4. Review CDN/proxy configuration
